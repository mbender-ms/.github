## Finding 1: NVA MTU requirement not mentioned

**Severity:** High — directly addresses gap analysis question: *"Azure load balancer to NVA asymmetric traffic issue?"* and *"Issue with Cisco FTDv with Azure Gateway Load Balancer"*

### What changed

- **Article:** Mentions VXLAN protocol is used, but does not mention MTU requirements for NVA virtual machines
- **Official best practices documentation:** Explicitly requires NVA MTU to be increased to a minimum of 1550 bytes (or up to 4000 for jumbo frames) to prevent packet drops from VXLAN headers
- **Status:** Content gap — missing a required configuration step that causes packet drops and NVA connectivity failures

### Why this matters

When VXLAN headers are added to packets by Gateway Load Balancer, the effective payload capacity decreases. If the NVA's MTU is left at the default 1500 bytes, packets are fragmented or dropped because the VXLAN-encapsulated packet exceeds the interface MTU. This is a top source of NVA failures in Gateway Load Balancer deployments and is likely the root cause of the gap analysis questions:
- "Azure load balancer to NVA asymmetric traffic issue?"
- "Issue with Cisco FTDv with Azure Gateway Load Balancer"
- "Azure Public load balancer packet forwarding to NVA"

The best practices article explicitly states: *"Ensure your NVAs MTU limit is increased to at least 1550, or up to the recommended limit of 4000 for scenarios where jumbo frames are used. Without increasing the MTU limit, you can experience packet drops due to the larger packet size from other packets generated by the VXLAN headers."*

### Evidence

- **Primary source:** https://learn.microsoft.com/azure/load-balancer/load-balancer-best-practices#implement-gateway-load-balancer-configuration-best-practices
  > "Ensure your NVAs MTU limit is increased to at least 1550, or up to the recommended limit of 4000 for scenarios where jumbo frames are used. Without increasing the MTU limit, you can experience packet drops due to the larger packet size from other packets generated by the VXLAN headers."
- **Tutorial source:** https://learn.microsoft.com/azure/load-balancer/tutorial-create-gateway-load-balancer
  > "If leveraging your own custom network virtual appliance in the backend pool of a Gateway Load Balancer, please ensure the MTU of all NVA virtual machines are raised to a minimum of 1550 bytes to accommodate for the VXLAN encapsulated headers."
- **Last verified:** April 30, 2026

### Proposed fix

In the **Data path diagram** section (after the paragraph explaining VXLAN encapsulation), add:

```markdown
> [!IMPORTANT]
> NVA virtual machines in the Gateway Load Balancer backend pool must have their MTU increased to a minimum of **1550 bytes** to accommodate VXLAN encapsulated headers. For environments using jumbo frames, the recommended MTU limit is 4000 bytes. Without this change, packet drops can occur because VXLAN headers increase packet size beyond the default 1500-byte MTU limit.
```

Alternatively, add a dedicated **NVA configuration requirements** subsection under **Components**:

```markdown
### NVA configuration requirements

NVA virtual machines in the Gateway Load Balancer backend pool require specific configuration to function correctly:

- **MTU requirement**: Increase the NVA MTU to a minimum of 1550 bytes to handle VXLAN-encapsulated packet headers. For jumbo frame scenarios, increase up to 4000 bytes. Leaving the default 1500-byte MTU results in packet drops and traffic failures.
- **Tunnel interface separation**: Use the **External** tunnel interface type for incoming (uninspected) traffic and the **Internal** tunnel interface type for outgoing (inspected) traffic. This separation ensures isolation of trusted and untrusted traffic.
```

## Finding 2: NVA asymmetric traffic — guidance not actionable

**Severity:** Medium

### What changed

- **Article:** States that Gateway Load Balancer "maintains flow stickiness to a specific instance in the backend pool along with flow symmetry" and that "packets traverse the same network path in both directions"
- **Official docs:** Confirm this behavior
- **Gap:** The article explains *what* Gateway Load Balancer does to prevent asymmetric traffic but does not explain what to do when asymmetric traffic occurs anyway (misconfigured UDR, chaining errors)
- **Status:** Accurate but incomplete for troubleshooting scenarios

### Why this matters

The gap analysis question "Azure load balancer to NVA asymmetric traffic issue?" implies users encounter asymmetric routing even after setting up Gateway Load Balancer. Common causes include:
- Using UDRs alongside Gateway Load Balancer (not supported — the article correctly notes UDRs are not needed)
- Incomplete chaining configuration
- MTU issues causing partial packet loss on one direction

The article already correctly states: *"Configuring a Gateway Load Balancer's frontend IP as the next hop in user-defined routes (UDRs) is not supported."*

### Evidence

- **Primary source:** https://learn.microsoft.com/azure/load-balancer/gateway-overview#configuration-and-supported-scenarios
- **NVA high availability patterns:** https://learn.microsoft.com/azure/architecture/networking/guide/network-virtual-appliance-high-availability#gateway-load-balancer

### Proposed fix

Expand the UDR NOTE in the **Configuration and supported scenarios** section:

```markdown
> [!NOTE]
> Configuring a Gateway Load Balancer's frontend IP as the next hop in user-defined routes (UDRs) is not supported. Gateway Load Balancer resources must be chained/referenced by a supported consumer resource such as a Standard Public Load Balancer or Standard NIC IP configuration. Using UDRs alongside Gateway Load Balancer can cause asymmetric traffic routing and NVA failures.
```
